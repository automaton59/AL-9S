记忆系统设计

AI 女友需要记住和你的对话历史、偏好、重要事件，这是"人格连续性"的关键

A

短期记忆 + 向量数据库检索

原理：最近对话放上下文，历史对话用向量数据库存储，对话时检索相关记忆

当前对话 →
最近 10 轮对话（上下文窗口）
历史记忆检索 →
向量数据库查询相关记忆：
- "你喜欢吃火锅"（3天前）
- "你妈妈叫李梅"（1周前）
- "你在北京工作"（2周前）
合并后送给 LLM →
系统提示 + 相关记忆 + 最近对话

优势

  • 可扩展性强，能存储无限历史
  • 检索准确，不会遗忘重要信息
  • 节省上下文开销
  • 业界主流方案（如 Character.AI）

劣势

  • 需要额外的向量数据库（Pinecone/Qdrant/ChromaDB）
  • 检索质量依赖 embedding 模型
  • 实现复杂度较高

技术栈：PostgreSQL + pgvector 或 Qdrant + OpenAI embedding

B

纯上下文窗口（简单粗暴）

原理：把所有对话历史直接塞进上下文窗口，不做检索

每次对话 →
系统提示
+ 所有历史对话（从第 1 天到现在）
+ 当前问题
限制 →
超过上下文长度后，丢弃最早的对话

优势

  • 实现极其简单
  • 不需要额外数据库
  • 记忆连贯，AI 能看到完整对话流
  • 适合 MVP 快速验证

劣势

  • 受上下文长度限制（Deepseek 最多 64K tokens）
  • 成本随对话增长线性增加
  • 超出后会"失忆"

技术栈：数据库存对话，每次拼接上下文

C

分层记忆（短期+长期+核心记忆）

原理：模拟人类记忆机制，分为短期、长期和核心记忆

核心记忆（永久）
你叫张伟 | 你妈妈叫李梅 | 你在北京工作程序员
长期记忆（重要事件，向量检索）
上周和女友吵架 | 去年去过日本旅游 | 喜欢吃火锅
短期记忆（最近对话）
今天聊了工作压力 | 刚才说想吃烧烤

优势

  • 最符合人类记忆模型
  • 核心信息永不丢失
  • 记忆质量最高
  • 上下文使用最高效

劣势

  • 需要 AI 自动分类记忆（或用规则）
  • 实现最复杂
  • 需要定期"记忆整理"

技术栈：结构化数据库 + 向量数据库 + 记忆分类系统

D

定期总结压缩

原理：每隔一段时间让 AI 总结对话，用总结替代原始对话

原始对话（50 轮）→
"今天天气真好" "是啊，想出去玩" "去哪呢"...
（省略 47 轮）
AI 总结 →
用户今天心情不错，想出去玩，最后决定去公园，
聊到了他最近工作压力大，我安慰了他。

优势

  • 实现相对简单
  • 上下文压缩效率高
  • 不需要向量数据库
  • 成本可控

劣势

  • 总结可能丢失细节
  • "我昨天说过什么"类问题难以回答
  • 总结质量依赖 LLM

技术栈：数据库 + 定时总结任务

我的推荐：分阶段实现

第一阶段（MVP）：方案 B（纯上下文）

  • 实现最简单，专注核心功能
  • Deepseek 64K 上下文够用几周

第二阶段：升级到方案 C（分层记忆）

  • 核心记忆用结构化字段（用户名、偏好、关系等）
  • 长期记忆用向量检索
  • 短期记忆保持纯上下文

方案 D 可以作为过渡方案，在方案 B 上下文快满时启用
