# AL-1S 项目背景与路线图

最后更新：2026-06-19

## 1. 项目定位

AL-1S 是一个本地优先的私人 IM 陪伴应用。

它的核心体验不是“配置一个 AI 工具”，而是“打开一个熟悉的聊天对象”。用户应该看到联系人、头像、在线状态、正在输入、聊天气泡、历史会话，以及一个逐渐稳定下来的关系对象。

AL-1S 不以 SillyTavern 式剧情玩法为目标，也不追求把世界书、设定集、复杂剧情结构搬进来。角色资料、记忆、人格稳定都应服务于更真实的长期相处，而不是服务于跑剧情。

## 2. 产品原则

### 2.1 像真人 IM

默认体验应接近 QQ、微信或其他私人聊天工具：

- 头像旁显示在线状态。
- 用户发送后显示“正在输入...”。
- 回复完成后整条出现，不默认做逐字流式输出。
- 用户可以撤回仍在发送中的消息，并打断请求。
- 错误状态要清楚，但不要把底层 API 调试感暴露给普通聊天。

### 2.2 不展示思维过程

AL-1S 的聊天对象不应把 thinking block 或 reasoning 作为聊天内容展示。底层可以解析 reasoning，用于兼容部分模型响应格式或未来调试，但普通用户体验中不显示、不强调，也不把它作为角色说出的话保存。

### 2.3 情绪是元数据，不是正文

模型可以输出类似 `[happy]` 或 `[emotion: happy]` 的标签，但前端应从正文中剥离。情绪只作为 metadata 保存，用于后续头像、表情、Live2D、视频聊天或状态变化。

### 2.4 本地优先

用户的角色、聊天、记忆、关系资料、API profile 默认保存在本机。应用可以调用用户配置的第三方 LLM API，但不应把 AL-1S 自己设计成强依赖云端账号和云端数据库的产品。

### 2.5 API 中立

AL-1S 应尽量兼容 OpenAI-compatible API。用户可以使用 OpenAI、DeepSeek、OpenRouter、AnyRouter、OpenCode 或其他兼容端点。具体 provider 的差异应尽量被 profile 和 adapter 消化，而不是散落在 UI 和业务代码里。

### 2.6 质量每阶段都做

测试、构建、迁移、安全、文档不是最后一个 Phase 才做的“收尾”。每一个阶段结束时都应该跑一轮完整检查，避免功能越堆越难维护。

## 3. 当前 Phase 1：私人 IM MVP

Phase 1 已完成，目标是把项目从基础 Demo 推到可日常试用的桌面聊天应用。

已完成能力：

- Electron + Vite 开发启动流程。
- API profile 管理：新建、保存、切换、删除、测试连接、获取模型。
- API key 与 profile 配对保存。
- Electron 下使用 `safeStorage` 加密 API key。
- 本地 JSON 文件存储：配置、聊天历史、API profiles。
- 浏览器开发模式 localStorage fallback。
- 多角色、多会话。
- 会话绑定角色。
- 消息编辑、撤回打断、重新生成、删除后续。
- 默认完整回复，不做逐字流式展示。
- 主聊天隐藏 thinking/reasoning。
- 情绪标签从正文剥离，仅保留 metadata。
- IM 风格 UI。
- 独立设置页。
- Light / Dark / System 主题。
- 自定义下拉菜单，避免原生 select 在暗色模式下割裂。

Phase 1 的意义是打通基础链路：用户能配置 API，能稳定聊天，数据能在本机保存，聊天体验已经从“模型调试窗口”转向“私人聊天空间”。

## 4. Phase 2：人格、关系、记忆与角色资产

Phase 2 是下一阶段主线。

它合并了原本可能拆开的“角色管理”和“记忆系统”。原因很简单：角色资料、人设稳定、长期记忆、关系档案、上下文装配，本质上都在解决同一个问题：让 AL-1S 在长期聊天中像同一个人，而不是每次对话都重新生成一个临时人格。

### 4.1 数据层先评估，不预设答案

SQLite 或更可靠的本地数据层值得评估，但不应在路线图里写死。JSON 不是完全不行，尤其在本地优先、可读、易备份、数据规模尚可控的阶段，它仍然可能是合理方案。

Phase 2 需要先回答这些问题：

- 消息和记忆数量会增长到什么规模？
- 是否需要复杂查询、索引、事务？
- 是否需要原子写、自动备份、崩溃恢复？
- 用户是否需要能直接看懂、手动修复、手动备份数据？
- 未来移动端是否要共享这套数据抽象？
- 当前 Phase 1 JSON 数据如何迁移？

候选方向：

- 继续 JSON，但补上 schema version、原子写、备份、校验和迁移。
- 使用 SQLite，获得事务、索引和更强查询能力。
- 混合方案：配置和 profile 保持 JSON，消息、记忆、索引进入 SQLite 或其他本地数据库。

Phase 2 的正确起点不是“装 SQLite”，而是“定义数据模型、迁移策略和可靠性标准”。

### 4.2 角色资料应像联系人档案

角色资料不应该变成设定集堆叠。它更像联系人档案和人格档案：

- 角色基本信息：姓名、头像、称呼、简介。
- 人格核心：说话方式、价值观、边界、情绪表达习惯。
- 关系基调：亲近程度、互动方式、重要共同经历。
- 资源索引：头像、未来表情、Live2D、视频聊天素材。
- Prompt 字段：哪些长期固定，哪些按上下文动态装配。

角色卡导入导出可以做，但内部格式应以 AL-1S 的长期相处目标为核心。兼容常见角色卡字段可以是实用功能，但不把 SillyTavern 完整兼容当主线。

### 4.3 记忆系统要可控

记忆不能只是“把所有旧聊天都塞进 prompt”。Phase 2 需要建立可管理的记忆层：

- 用户资料记忆。
- 关系记忆。
- 偏好记忆。
- 事实记忆。
- 临时上下文。
- 会话总结。
- 用户可编辑、删除、置顶、禁用的记忆管理界面。
- 记忆来源追溯。
- 上下文装配和裁剪策略。

记忆系统必须允许用户纠正。长期陪伴体验里，错误记忆比没有记忆更伤体验。

### 4.4 导入、导出、备份、恢复

Phase 2 需要给数据可迁移性打基础：

- 角色资料导入导出。
- 聊天记录导出。
- 全量备份和恢复。
- 资源目录管理。
- API key 默认不进入备份，除非用户明确选择。
- 迁移失败时保留旧数据。

## 5. Phase 3：角色存在感与视频聊天

Phase 3 再进入更强的角色存在感。

这一阶段可以做：

- 情绪和状态管线。
- 头像和表情切换。
- Live2D 或其他动态形象。
- 视频聊天式界面。
- TTS 与语音输入。
- 角色当前状态、氛围和动作反馈。

Phase 1 已经保留了 `emotion` metadata，Phase 2 也应继续维护资源和状态字段，为 Phase 3 铺路。

## 6. Phase 4+：功能池

Phase 4 之后不固定成某一个单一大主题，而是根据产品实际使用情况挑选功能。

候选方向：

- 正式打包和发布：exe、dmg、AppImage。
- 移动端聊天：PWA、Capacitor、React Native 或原生 APK 再评估。
- 工具调用：天气、时间、日程、搜索、本地文件、网页信息。
- 主动聊天：可控、低打扰、有免打扰和频率限制。
- 插件或技能系统。
- 多端同步或云备份，但默认仍本地优先。
- 更细的隐私、安全、成本和 token 使用控制。

## 7. 明确不做或暂不做

这些内容不是当前主线：

- 世界书、设定集、剧情书式玩法。
- 把 SillyTavern 完整兼容作为核心目标。
- 在主聊天里显示 thinking block。
- 默认逐字流式输出。
- 先做大而全的插件市场或角色社区。
- 在没有稳定数据模型前做大量资产和记忆功能堆叠。

这些不是永远不能做，而是它们不应挤占 AL-1S 当前最重要的目标：建立一个更真实、稳定、可长期相处的私人聊天对象。

## 8. 每阶段固定收尾

每个 Phase 结束前都应至少完成：

- `npm run build`
- `npm run electron:compile`
- 核心聊天链路手动回归。
- API profile 新建、切换、删除、重启验证。
- 数据迁移和备份验证。
- 亮色、暗色、窄屏 UI 检查。
- 搜索真实 API key，确认没有进入仓库。
- 更新 README、交接文档和路线图。
